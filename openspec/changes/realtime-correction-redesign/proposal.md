## Why

当前系统的「实时模式」仅支持录制完成后再进行离线分析，无法满足论文要求中「实时反馈延迟 ≤300ms」的性能指标。用户需要的是**边做动作边获得矫正提示**的体验，而不是录完再看报告。同时，视频分析 Tab 目前只支持本地上传，缺少摄像头录制后分析的功能。

## What Changes

- **重新设计实时模式 Tab**：从「录制→分析」模式改为「实时跟做→即时反馈→总体评分」模式
  - 选择动作类型后，3 秒倒计时准备
  - 开始跟做时，每帧进行关节角度计算 + 规则引擎检查，实时在旁边刷新矫正建议（延迟 ≤300ms）
  - 录制结束后，用完整序列做 DTW 对比，生成总体评分和总体建议报告
- **增强视频分析 Tab**：增加摄像头录制方式
  - 除本地上传外，新增「摄像头录制」子模式，录制完成后走与本地视频相同的分析流程
- **新增实时反馈引擎**：基于固定窗口 DTW 的轻量级实时分析模块
  - 每帧取最近 10-15 帧，用固定窗口 DTW 与模板对应进度的帧段做对比
  - 结合角度偏差 + 规则引擎给出即时文字提示（如「膝盖内扣，请外展」「背部弯曲，请挺直」）
  - 固定窗口保证计算量恒定（10×10 = 100 次距离计算，~5-10ms），满足 300ms 延迟要求
- **DTW 算法可选切换**：UI 中新增算法选择下拉框，支持四种模式
  - 全部使用经典 DTW
  - 全部使用 FastDTW
  - 全部使用 DerivativeDTW
  - 系统自动选择（实时用 FastDTW，离线用 DDTW）
  - 方便用户测试不同算法性能差异，为论文提供对比数据

## Capabilities

### New Capabilities
- `realtime-feedback`: 实时矫正反馈引擎 — 基于固定窗口 DTW + 角度偏差的即时检测与文字提示，延迟 ≤300ms
- `camera-recording`: 摄像头录制模式 — 视频分析 Tab 下新增摄像头录制方式，录制后走标准分析流程
- `algorithm-selector`: DTW 算法切换 — UI 中四选一（DTW / FastDTW / DDTW / 自动），支持用户测试对比

### Modified Capabilities
- `web-ui`: 重新设计实时模式 Tab 的交互流程（倒计时→实时跟做→即时反馈→总体报告），增强视频分析 Tab 支持摄像头录制

## Impact

- **修改文件**: `src/app/gradio_ui.py`（UI 布局和回调重写）、`src/app/pipeline.py`（新增实时分析方法）、`src/app/session.py`（新增实时反馈状态管理）
- **新增文件**: `src/correction/realtime_feedback.py`（实时反馈引擎，基于角度偏差 + 规则引擎的单帧/窗口分析）
- **依赖**: 无新增外部依赖，复用现有的 `AngleCalculator`、`CorrectionRuleEngine`、`TemplateLibrary`
- **性能约束**: 单帧分析延迟 ≤300ms（MediaPipe ~30-50ms + 角度计算 ~2ms + 规则引擎 ~2ms + Gradio 网络 ~100ms = ~150ms，满足要求）
- **预计开发时间**: 约 4-6 小时
